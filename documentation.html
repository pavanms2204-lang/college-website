01 · Project Overview

Overview

Project TitleSEA College of Engineering and Technology — Official Website
Website CategoryEducational Institution / College Website
Problem StatementProspective students, parents, and staff need a single, credible, mobile-friendly source of truth for the college's departments, admissions process, and campus life — replacing scattered PDFs and word-of-mouth information.
Why this project?A college website is a real-world brief with a genuine information architecture problem: multiple audiences (applicants, current students, faculty), multiple content types (academic, procedural, promotional), and a hard requirement for clarity and trust. It let us apply every mandatory technology — semantic HTML, responsive CSS, and interactive JavaScript — against a purposeful, non-trivial use case rather than a toy demo.
02 · Project Description

Description

Purpose. The website serves as the digital front door for SEA College of Engineering and Technology, communicating what the college teaches, how to apply, and what campus life looks like — while carrying a consistent visual identity across every page.

Target users. The primary audience is prospective undergraduate applicants and their families researching departments and the admissions timeline. Secondary audiences include current students looking up campus facilities and contact details, and evaluators reviewing this project for the competition.

Key features. The site spans seven pages — Home, About, Academics, Admissions, Campus Life, Contact, and this Documentation page — sharing a single responsive design system. It includes a searchable department directory, a validated multi-field admissions enquiry form with local-storage draft-saving, an FAQ accordion, a dark/light theme switcher persisted across visits, scroll-triggered reveal animations, and a fully responsive layout tested down to mobile widths.

Future improvements. Planned next steps include wiring the forms to a real backend or form-handling service, adding a student/faculty login portal, a searchable events calendar, and an image gallery once official campus photography is available.

06 · Challenges Faced

Challenges Faced

Keeping a shared header/footer consistent across seven static pages. Without a templating engine, every navigation change had to be applied by hand across all files. We solved this by finalising the navigation structure early and treating the header/footer markup as a fixed "component" copied identically into each page.

Form validation without a library. Writing real-time validation (required fields, email/phone format, minimum lengths) in plain JavaScript required careful event handling on blur and submit, plus keeping error states in sync with input changes.

Avoiding a generic look. Early drafts defaulted to a generic corporate template. We deliberately committed to a technical "blueprint / drafting" visual language — grid backgrounds, dimension lines, drafting-compass motifs — to make the design specific to an engineering college rather than interchangeable with any other institution.

Lesson learned: constraints (HTML/CSS/JS only, no frameworks) forced closer attention to semantic structure and CSS architecture than a framework would have — and made the final result easier to reason about end-to-end.

08 · Feedback on Frontend Web Development Training

Training Feedback

The training struck a good balance between fundamentals and practical application — rather than treating HTML, CSS, and JavaScript as separate modules, the sessions pushed us to combine all three on a real, non-trivial project from day one, which made the learning stick. The "no frameworks or libraries" rule felt restrictive at first, but it turned out to be the most valuable constraint of the whole competition: writing our own form validation, theme switcher, and scroll animations in plain JavaScript forced a much deeper understanding of the DOM than a library would have allowed.

If anything, we'd have liked a bit more guidance earlier on structuring CSS across multiple pages without a build system, since that's where most of our early time went. Overall, this was a demanding but genuinely rewarding two weeks, and it gave our team real confidence in shipping a complete, deployed website from scratch.
